# Server Backend Deployment Guide — Galala IAAS

> **Laravel 12 + Filament 3 + AI Chatbot Integration**
> Last updated: 2026-06-17

---

## 1. Project Purpose

The **Galala IAAS Backend** is a Laravel 12 application serving the Galala University Intelligent Access and Administration System. It provides:

- **Student Authentication** — Sanctum-based token auth for students.
- **Vehicle Access Management** — Students submit vehicle permit requests; admins approve/reject; a gate API verifies plate numbers via OCR.
- **AI Chatbot** — Students and guests can chat with a Galala-specific AI assistant. Student chats are persisted in MySQL; guest chats live in Redis with a configurable TTL.
- **Filament Admin Panel** — Super-admin and support-admin interface for managing students, faculties, vehicle requests, and viewing chat conversations.

## 2. What Changed (Recent Backend Additions)

The following features were added as part of the AI chatbot integration phases:

| Feature | Description |
|---|---|
| Student Chat API | Full CRUD + message flow via `api/v1/student/chats/...` |
| Guest Chat API | Stateless Redis-only guest chat via `api/v1/guest/chat/...` |
| Redis AI Queue | AI chat jobs dispatched onto `ai-chat` Redis queue |
| AI HTTP Client | `HttpAiChatClient` → FastAPI AI service at `/api/chat` and `/api/chat/summarize` |
| `chat_conversations.session_id` | Links student MySQL conversations to AI-side Redis session |
| `chat_summaries` table | AI service writes summaries here via `/api/chat/summarize` |
| `ai_chat_messages` DB view | Read-only view the AI service reads for conversation history |
| Filament Chat Viewer | Admin can view, soft-restore, and hard-delete chat conversations |
| Student Deletion Protection | Cannot delete a student who has any chat history |
| `chat:summarize-idle` command | Scheduled command dispatches summarization for idle student chats |

## 3. Runtime Services Required

| Service | Role |
|---|---|
| **Laravel PHP-FPM app** | Handles HTTP requests (via nginx) |
| **nginx** | Web server / reverse proxy for the Laravel app |
| **MySQL 8** | Primary database for all persistent data |
| **Redis 7** | Queue backend for AI chat jobs; guest chat session storage; app cache |
| **Laravel queue worker** | Processes the `ai-chat` queue (runs `ProcessStudentAiChat` and `ProcessGuestAiChat` jobs) |
| **Laravel scheduler** | Runs `chat:summarize-idle` every 10 minutes to trigger AI summarization |
| **FastAPI AI Service** | External AI service that handles `/api/chat` and `/api/chat/summarize` |
| **Redis (AI side)** | AI service's own Redis for guest session memory and AI session state |
| **Ollama (AI side)** | LLM serving on the AI side (managed by the AI team) |

> **Note**: The Laravel queue worker and scheduler must run as separate long-lived processes (Docker containers or supervised services), not as part of the PHP-FPM/nginx request lifecycle.

## 4. Required Laravel Environment Variables

All variables below must be set in the server's `.env` file (copy from `.env.example` as a starting point):

```env
# Application
APP_NAME="Galala IAAS"
APP_ENV=production
APP_KEY=<generated by: php artisan key:generate>
APP_DEBUG=false
APP_URL=https://your-server-domain.example.com

# Database
DB_CONNECTION=mysql
DB_HOST=mysql
DB_PORT=3306
DB_DATABASE=iaas_db
DB_USERNAME=iaas_user
DB_PASSWORD=<strong password>

# Redis (queue + cache + guest sessions)
QUEUE_CONNECTION=redis
CACHE_STORE=redis
REDIS_CLIENT=phpredis
REDIS_HOST=redis
REDIS_PASSWORD=<strong secret>
REDIS_PORT=6379
REDIS_QUEUE_RETRY_AFTER=510

# Gate API
GATE_API_KEY=<strong secret — must match the gate hardware config>

# AI Service
AI_CHAT_DRIVER=http
AI_CHAT_CONNECTION=redis
AI_CHAT_QUEUE=ai-chat
AI_SERVICE_URL=http://127.0.0.1:8010
AI_SERVICE_TOKEN=<must match AI service AI_SERVICE_TOKEN>
AI_CHAT_PATH=/api/chat
AI_CHAT_SUMMARIZE_PATH=/api/chat/summarize
AI_CONNECT_TIMEOUT_SECONDS=10
AI_RESPONSE_TIMEOUT_SECONDS=420

# Chat behaviour
CHAT_MAX_MESSAGE_LENGTH=3000
GUEST_SESSION_TTL_SECONDS=7200
GUEST_PENDING_TTL_SECONDS=600
CHAT_SUMMARIZE_IDLE_SECONDS=7200
GUEST_MESSAGE_THROTTLE_REQUESTS=10
GUEST_MESSAGE_THROTTLE_MINUTES=1
```

> **Critical**: `AI_CHAT_DRIVER=http` must be set. The default is `fake` which does not call the AI service.

## 5. AI Service Environment Relationship

```
┌─────────────────────────────┐        Bearer <AI_SERVICE_TOKEN>       ┌──────────────────────────────┐
│   Laravel Backend           │  ─────────────────────────────────────>│   FastAPI AI Service         │
│   AI_SERVICE_URL points to  │        POST /api/chat                   │   Listens on port 8010       │
│   AI host:port 8010         │        POST /api/chat/summarize         │   AI_SERVICE_TOKEN must match│
└─────────────────────────────┘                                         └──────────────────────────────┘
```

- `AI_SERVICE_URL` in Laravel `.env` must point to the FastAPI AI service's host and port.
- `AI_SERVICE_TOKEN` in Laravel `.env` **must exactly match** the `AI_SERVICE_TOKEN` configured in the FastAPI AI service.
- On our server, port 8001 is already in use by another FastAPI service. The Galala AI chatbot service must be configured to listen on **host port 8010**.
- If both run in Docker Compose, use the Compose service name: `http://ai-service:8000` (or whichever internal port the AI service listens on).

## 7. Database Tables and Views Created

| Name | Type | Description |
|---|---|---|
| `chat_conversations` | Table | One row per student chat thread; contains `uuid`, `session_id`, `title`, `status`, `last_message_at`, soft-delete fields |
| `chat_messages` | Table | Individual messages (user/assistant) with `uuid`, `role`, `content`, `status`, `sequence_number`, `client_message_id` |
| `chat_ai_requests` | Table | Tracks each queued AI call; links user message → assistant message; stores `status`, `error_code`, `attempt_number` |
| `chat_summaries` | Table | Written by the AI service via `/api/chat/summarize`; keyed by `user_id` + `session_id` |
| `ai_chat_messages` | View | Read-only view the AI service uses to fetch conversation history; excludes in-flight turns |

### `ai_chat_messages` view columns:
```
session_id, user_id, role, content, created_at
```

## 8. Student Chat Flow

```
Frontend                    Laravel Backend                     AI Service
   │                              │                                  │
   │ POST /api/v1/student/chats   │                                  │
   │ {message, client_message_id} │                                  │
   │ ─────────────────────────>   │                                  │
   │                              │ Create conversation row          │
   │                              │ Create user message (completed)  │
   │                              │ Create assistant message (pending)│
   │                              │ Create ai_request (queued)       │
   │                              │ Dispatch ProcessStudentAiChat    │
   │ 202 {chat.uuid,              │  on redis/ai-chat queue          │
   │      user_message.uuid,      │                                  │
   │      assistant_message.uuid, │                                  │
   │      ai_request.status}      │                                  │
   │ <─────────────────────────   │                                  │
   │                              │                                  │
   │ [polling: every 2-5s]        │                                  │
   │ GET .../messages/{uuid}/status│                                 │
   │ ─────────────────────────>   │                                  │
   │                              │ Worker picks up job              │
   │                              │ POST AI /api/chat                │
   │                              │ {user_id, message, session_id?}  │
   │                              │ ─────────────────────────────>   │
   │                              │ ← {message, session_id, user_id} │
   │                              │ Save session_id (first turn)     │
   │                              │ Update assistant message         │
   │                              │  content + status=completed      │
   │                              │ Update ai_request=completed      │
   │ 200 {status: "completed",    │                                  │
   │      content: "...answer..."}│                                  │
   │ <─────────────────────────   │                                  │
```

### Key points:

- **`client_message_id`**: A UUID the frontend generates per message. Re-sending with the same `client_message_id` returns the existing result (idempotency). Never reuse it across different messages.
- **`session_id`**: Generated by the AI service on the first turn; stored in `chat_conversations.session_id`. All subsequent turns for the same conversation send this `session_id` to the AI. The AI also reads history from the `ai_chat_messages` view using this key.
- **409 Conflict**: Returned if a previous message for this conversation is still pending.
- **Retry**: If an assistant message is in `failed` status, the frontend can call `POST .../messages/{messageUuid}/retry` to re-queue.
- **Soft delete**: `DELETE /api/v1/student/chats/{chatUuid}` sets `deleted_by_student_at`; the conversation is hidden from the student but remains in the database for admin review.

## 9. Guest Chat Flow

```
Frontend                    Laravel Backend                     AI Service
   │                              │                                  │
   │ POST /api/v1/guest/chat/messages│                              │
   │ {message}  [no token yet]    │                                  │
   │ ─────────────────────────>   │                                  │
   │                              │ Generate guest_token (64 chars)  │
   │                              │ Hash token → tokenHash           │
   │                              │ Create Redis requestId           │
   │                              │ Append user message to Redis     │
   │                              │ Dispatch ProcessGuestAiChat      │
   │ 202 {request_id,             │  on redis/ai-chat queue          │
   │      guest_token,            │                                  │
   │      status: "queued"}       │                                  │
   │ <─────────────────────────   │                                  │
   │                              │                                  │
   │ Store guest_token in memory  │                                  │
   │ or sessionStorage            │                                  │
   │                              │                                  │
   │ [polling] GET .../messages/{requestId}/status                   │
   │ X-Guest-Token: <token>       │                                  │
   │ ─────────────────────────>   │                                  │
   │                              │ Worker picks up job              │
   │                              │ POST AI /api/chat                │
   │                              │ {user_id: 0, message, session_id?}│
   │                              │ ─────────────────────────────>   │
   │                              │ ← {message, session_id, user_id} │
   │                              │ Save ai_session_id in Redis      │
   │                              │ Complete request in Redis        │
   │ 200 {status: "completed",    │                                  │
   │      content: "...answer..."}│                                  │
   │ <─────────────────────────   │                                  │
   │                              │                                  │
   │ [follow-up message]          │                                  │
   │ POST .../messages            │                                  │
   │ X-Guest-Token: <token>       │                                  │
   │ {message}                    │                                  │
   │ ─────────────────────────>   │                                  │
```

### Key points:

- **`guest_token`**: 64-character alphanumeric random string. Only returned on the **first** message. Store it and send it in `X-Guest-Token` header for all subsequent requests.
- **No database storage**: Guest conversations live in Redis only. They expire after `GUEST_SESSION_TTL_SECONDS` (default: 7200 seconds = 2 hours) of inactivity.
- **`user_id = 0`**: Guest requests are sent to the AI with `user_id = 0`. The AI service manages guest memory by its own `session_id` (stored in Redis on the backend).
- **Status values**: `queued`, `processing`, `completed`, `failed`.
- **History**: `GET /api/v1/guest/chat/history` with `X-Guest-Token` returns all messages for the guest session. Also refreshes the TTL.

## 10. Summarization Flow

```
Scheduler (every 10 minutes)
  └─ chat:summarize-idle
       └─ Finds signed-in conversations that are:
            - Have session_id set
            - Idle > CHAT_SUMMARIZE_IDLE_SECONDS (default 2h)
            - No queued/processing AI work
            - No up-to-date summary in chat_summaries
       └─ Dispatches SummarizeChatConversation (unique per conversation)
            └─ POST AI /api/chat/summarize
                 {user_id, session_id}
               ← {status: "created"|"updated"|"skipped", summary_preview, session_id, user_id}
               AI writes/updates chat_summaries table row
```

> **Guests are never summarized.** They have no `chat_conversations` rows. The command inherently skips them.

## 11. Queue Worker Command

The worker **must** be running as a long-lived process for AI chat to work:

```bash
php artisan queue:work redis --queue=ai-chat --tries=1 --timeout=460 --sleep=3
```

| Flag | Value | Reason |
|---|---|---|
| `redis` | connection | Uses the `redis` queue connection |
| `--queue=ai-chat` | queue name | Must match `AI_CHAT_QUEUE` env var |
| `--tries=1` | max attempts | AI jobs are not retried; the `failed()` method handles failure cleanly |
| `--timeout=460` | seconds | Slightly above `AI_RESPONSE_TIMEOUT_SECONDS=420` to allow clean failure handling |
| `--sleep=3` | seconds | Poll interval when queue is empty |

In Docker Compose, the `worker` service handles this automatically (already configured in `docker-compose.yml`).

## 12. Laravel Scheduler

The scheduler must run every minute via cron or a supervised `schedule:work` process:

**Option A — Cron (server-level):**
```cron
* * * * * cd /var/www/html && php artisan schedule:run >> /dev/null 2>&1
```

**Option B — Supervised `schedule:work` (long-lived process):**
```bash
php artisan schedule:work
```

The scheduler runs `chat:summarize-idle` every 10 minutes with `withoutOverlapping()` to avoid concurrent runs.

## 13. Deployment Steps

### First-time deployment:

1. **Back up current server database** (if upgrading):
   ```bash
   mysqldump -u iaas_user -p iaas_db > backup_$(date +%Y%m%d_%H%M%S).sql
   ```

2. **Upload/pull new code** to the server:
   ```bash
   git pull origin main
   # OR: rsync / scp / deploy script
   ```

3. **Copy and update `.env`** carefully — do not overwrite existing secrets:
   - Start from `.env.example`
   - Preserve existing `APP_KEY`, `DB_PASSWORD`, `REDIS_PASSWORD`
   - Add new variables: `AI_CHAT_DRIVER`, `AI_SERVICE_URL`, `AI_SERVICE_TOKEN`, etc.
   - Set `AI_CHAT_DRIVER=http` (not `fake`)
   - Set `QUEUE_CONNECTION=redis`
   - Set `CACHE_STORE=redis`

4. **Install production dependencies** (no dev packages):
   ```bash
   composer install --no-dev --optimize-autoloader
   ```

5. **Build frontend assets** (if vite/npm assets changed):
   ```bash
   npm install && npm run build
   ```

6. **Clear old config/route caches**:
   ```bash
   php artisan config:clear
   php artisan route:clear
   php artisan cache:clear
   ```

7. **Run database migrations**:
   ```bash
   php artisan migrate --force
   ```
   This will create: `chat_conversations`, `chat_messages`, `chat_ai_requests`, `chat_summaries`, and the `ai_chat_messages` view.

8. **Rebuild optimized caches**:
   ```bash
   php artisan optimize
   # Equivalent to: config:cache + route:cache + view:cache + event:cache
   ```

9. **Restart containers/workers**:
   ```bash
   docker compose down && docker compose up -d
   # OR: supervisorctl restart all
   ```

10. **Verify health**:
    ```bash
    curl http://your-server/up
    # Expected: 200 OK
    ```

### Upgrade (subsequent deployments):

Same as above starting from step 2. Always back up DB first.

## 14. Rollback Plan

1. **Stop the worker** to prevent new jobs processing against the old code.
2. **Restore the previous code** version:
   ```bash
   git checkout <previous-tag-or-commit>
   ```
3. **Restore the database** from backup if migrations were destructive.
4. **Rollback migrations** if needed:
   ```bash
   php artisan migrate:rollback
   ```
5. **Restart containers/workers**.

> The chat-related migrations add new tables and a view — they do not modify existing tables (except adding `session_id` column to `chat_conversations`). Rolling back is safe.

## 15. Security Notes

| Risk | Mitigation |
|---|---|
| `.env` in version control | `.env` is in `.gitignore`. Never commit it. Use `.env.example` as template. |
| AI service token exposure | `AI_SERVICE_TOKEN` must be kept secret; it is never logged, never included in responses. |
| Guest token in logs | Guest tokens and their SHA-256 hashes are never logged. Only UUID request IDs appear in logs. |
| Authorization header in logs | The Bearer token sent to the AI is never logged. Only stable identifiers (UUIDs, error codes) are logged on failure. |
| Bulk hard-delete | Filament admin cannot bulk-hard-delete chat conversations. Only single hard-delete is available, and only when the conversation is not queued/processing. |
| Student deletion protection | Deleting a student via Filament admin is blocked if the student has any chat history (policy check + DB constraint). |
| Gate API key | `GATE_API_KEY` must be set to a strong random secret. The gate hardware must send this key in the `X-Gate-Api-Key` header. |
| SQL injection | All queries use Laravel Eloquent / query builder with parameterized queries. |
